## 一、测试方法论

### 1.1 测试原则

本测试方案采用**端到端真实验证**方法，而非隔离的单元测试，原因如下：

1. **真实环境验证**：通过MCP协议真实调用，验证完整链路的正确性
2. **全链路可观测**：观测从请求到响应的完整日志，定位问题根源
3. **场景驱动**：通过丰富的业务场景验证，而非简单的功能点测试
4. **边界探测**：深挖边界场景，发现设计盲区

### 2.3 结果判断标准

#### 召回率计算
```
召回率 = 实际召回的相关结果数 / 应该召回的相关结果总数 × 100%

目标: ≥ 75%
```

#### 精确率计算
```
精确率 = 实际召回的相关结果数 / 实际召回的结果总数 × 100%

目标: ≥ 80%
```

#### 近义词覆盖率
```
近义词覆盖率 = 通过近义词查询成功召回的次数 / 近义词查询总次数 × 100%

目标: ≥ 80%
```

- maxTotal（最大连接数）：200
- maxIdle（最大空闲连接）：50
- minIdle（最小空闲连接）：10

1. Kafka分区数不足，单分区消费能力有限

1. 增加Kafka Topic的分区数从3增加到12
2. 调整fetch.max.bytes从默认的52428800增加到104857600
3. 优化数据库查询，添加必要索引

效果：消费延迟恢复到毫秒级，TPS从500提升到3000。
```

通用配置原则：
- 最大连接数 = (CPU核心数 * 2) + 磁盘数
- 最小空闲连接 = 最大连接数 / 10
- 连接超时时间 = 业务RT * 3

1. MySQL连接池（HikariCP）：
   - maximumPoolSize: 20
   - minimumIdle: 2
   - connectionTimeout: 30000

2. Redis连接池（Jedis）：
   - maxTotal: 200

   - maxWaitMillis: 3000

3. MongoDB连接池：
   - maxPoolSize: 100
   - minPoolSize: 10

### 6.1 常见问题诊断流程

#### 问题1：实体被过度拆分

**现象**：

   → 查看实体列表是否包含完整的"Redis连接池"

2. 检查LLM响应：
   📄 [增强分析] LLM响应长度: X 字符
   → 分析JSON中entities数组的内容

3. 检查Prompt是否包含Few-Shot示例
```

**可能原因**：
- LLM提取Prompt不够精准
- Few-Shot示例不足
- 输入内容格式异常

**解决方向**：
- 优化Prompt中的语义完整性说明
- 增加边界情况的Few-Shot示例

---

#### 问题2：近义词无法召回

**现象**：
- 存储"Redis连接池"，查询"Jedis连接池"无结果
- 向量相似度分数过低

**诊断步骤**：
```
1. 检查Entity向量检索日志：
   🔍 [Entity向量检索] 找到实体: ...
   → 查看返回的实体列表和相似度分数

2. 检查相似度阈值：
   - 当前阈值设置是否过高（如0.8以上）
   - 实际相似度分数是多少

3. 检查embedding模型：
   - 确认embedding服务正常
   - 检查向量维度是否一致
```

**可能原因**：
- 相似度阈值设置过高
- Embedding模型对技术术语理解不足
- 实体向量未正确存储

**解决方向**：
- 适当降低相似度阈值（建议0.7）
- 考虑使用更适合技术领域的embedding模型

**可能原因**：
- neo4j_node_id未正确关联
- Neo4j存储失败

【技术方案】Redis连接池配置优化
在Spring Boot项目中使用Jedis作为Redis客户端，配置Redis连接池参数：
- maxTotal（最大连接数）：200
- maxIdle（最大空闲连接）：50
- minIdle（最小空闲连接）：10
- maxWaitMillis（最大等待时间）：3000ms

遇到的问题：高并发场景下Redis连接池耗尽，导致大量请求超时。
解决方案：将maxTotal从50调整到200，同时开启连接池预热。
```

#### D1测试数据
```
【项目记录】电商平台订单支付链路优化项目

项目背景：
用户在手机APP下单后，订单服务调用库存服务检查商品库存，然后通过消息队列通知
仓储系统进行预占。支付环节通过支付网关调用微信支付和支付宝完成资金结算，
支付成功后触发订单状态变更。物流系统接收发货指令，调用顺丰API生成运单，
通过短信服务通知用户。

技术栈：
- 订单服务(OrderService)：Spring Boot + MySQL
- 库存服务(InventoryService)：Spring Boot + Redis + MySQL
- 支付网关(PaymentGateway)：Spring Cloud Gateway
- 物流系统(LogisticsService)：Spring Boot + Elasticsearch
- 消息队列：RabbitMQ
- 缓存层：Redis Cluster
- 数据库：MySQL主从 + ShardingSphere分库分表

性能问题：
性能瓶颈出现在支付回调环节，平均响应时间从200ms增长到3秒。
经分析发现是数据库写锁竞争导致的，通过分库分表和读写分离解决了问题。

优化效果：
- 支付回调响应时间：3s → 200ms
- 订单创建TPS：500 → 3000
- 系统可用性：99.5% → 99.99%
