## 背景与思路

### 核心灵感来源

Context-Keeper的设计灵感很多源于**学生时代的学习方法论**,已经实现的两大能力:

1. **📅 时间线(Timeline)**: 按时间顺序整理笔记,复习知识脉络（比如文科类的历史、政治等学科）
   - AI化实现: TimelineRetriever - 按时序检索记忆
   - 核心价值: 理解事物的演进过程,把握发展规律

2. **🕸️ 知识图谱(Knowledge Graph)**: 构建知识体系,建立概念关联（比如理科类的化学、物理等学科）
   - AI化实现: KnowledgeGraphRetriever - 关系推理
   - 核心价值: 理解事物的关联关系,触类旁通

### 第三大能力: 错题本

基于同样的学习方法论哲学,提出第三大能力:

**📕 错题本(Error Book)**: 犯过的错误,确保不犯第二次
- AI化实现: ErrorBookRetriever - 负向经验召回
- 核心价值: 从失败中学习,避免重蹈覆辙
- **核心理念**: "人这一辈子,只要能做到同样的错误不犯第二次,就已经很牛逼了,比大部分人厉害"

### 三大能力的统一哲学

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Context-Keeper的设计哲学 = 人类学习方法论的AI化
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

深层哲学统一:
- 时间线(Timeline)     = 纵向维度(历史演进)
- 知识图谱(KG)         = 横向维度(空间关联)
- 错题本(Error Book)   = 深度维度(经验教训)

三者结合 = 完整的学习闭环!
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 与强化学习的异曲同工之妙

**强化学习的核心机制**:
- 正向奖励(Positive Reward): 鼓励成功行为,增加该行为的概率
- 负向惩罚(Negative Reward): 抑制失败行为,降低该行为的概率

**错题本的核心机制**:
- 正向记忆(Success Memory): 记录成功经验,作为未来参考
- 负向记忆(Error Memory): 记录失败教训,优先召回避免重犯

**映射关系**:
```
RL的Negative Reward  ≈  错题本的Error Memory
        ↓                        ↓
降低失败行为概率      优先召回失败场景,主动避开
```

**关键差异**:
- 强化学习(RL): 需要数千次试错,通过梯度下降调整参数,黑盒优化,需要GPU集群
- 错题本(Error Book): 一次记录立即生效,无需训练,透明可控,成本极低

**Context-Keeper的智慧**: **用"显式规则"实现"隐式学习"的效果,用"1%的成本"达到"80%的效果"**

## 核心问题分析

### 当前正向召回的痛点

**问题描述**: 正向排序往往比较经验主义,容易忽略重点场景、重大教训问题的场景

**具体案例**:

```
场景: 用户问"如何优化数据库查询?"

当前正向召回(按语义相似度排序):
1. [相似度0.92] 上次优化成功的SQL案例 ✅
2. [相似度0.88] 数据库索引配置经验 ✅
3. [相似度0.85] 查询优化通用技巧 ✅
4. [相似度0.78] 一个月前的失败教训 ⚠️
   → 因为相似度低,排在第4位,容易被忽略!

问题根源:
- 失败教训的embedding相似度往往不高
- 描述失败的用词与描述成功的用词不同
- 导致失败经验被排在后面,容易被忽略!
```

**真实案例**:

```
历史失败记录:
"上次在订单表上加索引时,因为没有考虑锁表问题,
 导致生产环境阻塞了2小时,紧急回滚..."

当前查询:
"帮我在订单表上加个索引优化查询"

正向召回结果(按语义相似度):
1. 索引优化成功案例(相似度0.91) ✅
2. 索引类型选择指南(相似度0.87) ✅
3. **失败教训**(相似度0.75) ⚠️ ← 排在第3,易忽略!

后果:
LLM看到前2条成功案例,信心满满地建议加索引,
完全没注意到第3条的失败教训,再次踩坑!
```

**核心问题**: **正向排序的"经验主义"导致失败教训被淹没在成功案例的海洋中!**

### Context-Keeper的独特优势

**MCP Server形态的天然优势**(相比独立Agent):

1. **实时感知对话**:
   - 每一轮对话都经过Context-Keeper
   - 能准确捕捉用户的负面反馈词("失败了"、"不行"、"出错了"、"重新来")
   - 实时检测到用户多次重试同一问题

2. **通过CursorRule主动介入LLM行为**:
   - 在LLM生成回复前,先召回错题本
   - 修改LLM的Prompt,强制其考虑历史教训
   - 实现"事前预防",而非"事后总结"

3. **透明可控,用户信任度高**:
   - 用户能看到错题本记录过程
   - 用户可以编辑/删除错题本条目
   - 用户理解"AI为什么这样建议"

**结论**: **MCP Server形态天然适合"错题本"能力!**

## 方案演进过程

### 方案讨论的关键问题

**问题1**: 是否需要单独实现一个【错题本】能力?还是融合到反馈策略的打分机制中?

**问题2**: 知识本身是静态的,如何评价其相关性?
- 跟这个query匹配度不高,不代表跟另一个query匹配度不高
- 如果只是简单打分,怎么跟场景进行关联?

**问题3**: 错题本是针对特定场景或领域的,如何做场景管理?

### 方案1: 统一打分机制(融合到反馈策略)

**核心思路**: 不单独建错题本,而是给所有记忆打"场景相关度 × 正负向权重"的分

```
记忆评分 = 语义相似度 × 正负向权重 × 新鲜度

其中:
- 正向记忆(成功经验): 权重 = +1.0
- 负向记忆(失败教训): 权重 = +2.0 (更高优先级)

检索时:
1. 先计算语义相关度(动态,与query相关)
2. 再乘以正负向权重(静态,记忆固有属性)
3. 最终得分 = 相关度 × 权重
```

**优势**:
- ✅ 架构简单,不需要单独的ErrorBook存储
- ✅ 自动处理场景相关性(通过语义相关度)
- ✅ 统一的评分体系

**劣势**:
- ❌ **场景绑定不明确**: 无法显式标记"这是订单表加索引的教训"
- ❌ **触发模式缺失**: 无法主动检测"用户正在做危险操作"
- ❌ **预防措施丢失**: 只能召回记忆,但缺少结构化的预防建议
- ❌ **团队共享困难**: 很难筛选出"团队最重要的10条教训"

### 方案2: 专属错题本(单独存储+特殊召回)

**核心思路**: 单独建立ErrorBook存储,包含场景信息、触发模式、预防措施

**优势**:
- ✅ **场景绑定明确**: 显式存储触发场景和模式
- ✅ **主动检测能力**: 可以基于trigger_patterns主动预警
- ✅ **结构化预防措施**: 不仅召回教训,还提供明确的预防方案
- ✅ **团队共享友好**: 可以按severity筛选TOP-10展示给新人
- ✅ **复盘分析方便**: 单独的错题本数据,便于统计和分析

**劣势**:
- ❌ 需要额外的存储结构
- ❌ 需要单独的检索通道
- ❌ 架构复杂度略高

### 最终决策

**选择方案2(专属错题本)+ 方案1的动态评分思想**

**核心理由**:
1. 错题本不仅是"负向评分",更是"完整的学习案例"
2. 需要包含:背景、过程、错误、正解(四段式)
3. 需要主动检测能力(trigger_patterns)
4. 需要团队共享能力(按severity筛选)

## 最终设计方案

### 1. 整体架构: 五维并行召回

```
┌─────────────────────────────────────────────────────────┐
│  第一阶段: 五维并行宽召回                               │
│  ┌────────────────────────────────────────────────┐    │
│  │  1. VectorRetriever      → TOP-50 (语义)      │    │
│  │  2. TimelineRetriever    → TOP-30 (时序)      │    │
│  │  3. KGRetriever          → TOP-20 (关系)      │    │
│  │  4. ErrorBookRetriever   → TOP-10 (错题) 🆕  │    │
│  │  5. SuccessRetriever     → TOP-20 (成功案例)  │    │
│  │                                                │    │
│  │  并行执行,总候选集: ~130条                    │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  第二阶段: LLM两阶段精排序(Token优化)                   │
│                                                         │
│  阶段2.1: 精排序(传Summary)                             │
│  ┌────────────────────────────────────────────────┐    │
│  │  输入: 所有候选集的摘要(~100 tokens/条)       │    │
│  │  输出: 按相关性排序的TOP-N(例如TOP-10)        │    │
│  │  Token消耗: 130条 × 100 = 13,000 tokens       │    │
│  └────────────────────────────────────────────────┘    │
│                                                         │
│  阶段2.2: 最终合成(传完整内容)                          │
│  ┌────────────────────────────────────────────────┐    │
│  │  输入: TOP-N的完整内容                         │    │
│  │    - 错题本: 四段式完整结构(~800 tokens/条)   │    │
│  │    - 正常记忆: 完整内容(~400 tokens/条)       │    │
│  │  输出: 最终合成的回复                          │    │
│  │  Token消耗: 假设TOP-5,约3,000 tokens          │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

Token优化效果:
- 原方案(所有候选集传完整内容): ~28,000 tokens
- 优化方案(两阶段传递): ~16,000 tokens
- 节省: 43% ✅
```

### 2. 核心数据结构: 四段式ErrorBookEntry

基于讨论确定的**四段式完整结构**:

```go
type ErrorBookEntry struct {
    // === 核心标识 ===
    ErrorID    string    `json:"error_id"`
    Title      string    `json:"title"`
    
    // ========================================
    // 🔥 四段式完整结构
    // ========================================
    
    // 第1段: 背景(Background) - 当时的场景
    Background struct {
        Scenario        string    `json:"scenario"`         // "需要优化订单查询性能"
        InitialGoal     string    `json:"initial_goal"`     // "在订单表上添加索引"
        Context         string    `json:"context"`          // "生产环境,日订单量10万+"
        Constraints     []string  `json:"constraints"`      // ["24小时服务","不能停机"]
        TriggerTime     time.Time `json:"trigger_time"`     // 2024-01-15
        RelatedCode     []string  `json:"related_code"`     // 相关代码文件
        RelatedConcepts []string  `json:"related_concepts"` // ["数据库索引","DDL"]
    } `json:"background"`
    
    // 第2段: 过程(Process) - 当时做了什么
    Process struct {
        ApproachTaken   string   `json:"approach_taken"`   // "直接执行ALTER TABLE"
        StepByStep      []string `json:"step_by_step"`     // 具体步骤
        DecisionMaking  string   `json:"decision_making"`  // "以为表数据不多"
        Expectations    string   `json:"expectations"`     // "预期1分钟完成"
    } `json:"process"`
    
    // 第3段: 错误(Error) - 出了什么问题
    Error struct {
        WhatHappened    string    `json:"what_happened"`    // "表被锁,订单服务阻塞"
        RootCause       string    `json:"root_cause"`       // "未使用ONLINE DDL"
        Impact          string    `json:"impact"`           // "业务阻塞2小时"
        Severity        Severity  `json:"severity"`         // critical
        Symptoms        []string  `json:"symptoms"`         // ["响应超时","订单失败"]
        ErrorTime       time.Time `json:"error_time"`       // 具体时间
        RecoveryAction  string    `json:"recovery_action"`  // "紧急回滚"
        Losses          string    `json:"losses"`           // "估计损失50万"
    } `json:"error"`
    
    // 第4段: 正解(Solution) - 应该怎么做
    Solution struct {
        CorrectApproach      string   `json:"correct_approach"`      // "使用pt-online-schema-change"
        StepByStep           []string `json:"step_by_step"`          // 正确的步骤
        WhyItWorks           string   `json:"why_it_works"`          // "不阻塞业务查询"
        Prevention           []string `json:"prevention"`            // 预防措施列表
        BestPractices        []string `json:"best_practices"`        // 最佳实践
        AlternativeOptions   []string `json:"alternative_options"`   // 其他可行方案
        Checklist            []string `json:"checklist"`             // 操作前检查清单
    } `json:"solution"`
    
    // === 检索增强字段 ===
    TriggerPatterns []string  `json:"trigger_patterns"` // 触发模式
    Keywords        []string  `json:"keywords"`         // 关键词
    Embedding       []float64 `json:"embedding"`        // 向量(用于语义召回)
    
    // === 反馈闭环 ===
    RecurrenceCount int        `json:"recurrence_count"` // 重犯次数
    LastRecurrence  *time.Time `json:"last_recurrence"`  // 最后重犯时间
    IsResolved      bool       `json:"is_resolved"`      // 是否已彻底解决
    
    // === 元数据 ===
    SessionID   string    `json:"session_id"`
    UserID      string    `json:"user_id"`
    WorkspaceID string    `json:"workspace_id"`
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`
}

// Severity: 严重程度枚举
type Severity string
const (
    SeverityCritical Severity = "critical"  // 致命(生产故障)
    SeverityHigh     Severity = "high"      // 严重(重大bug)
    SeverityMedium   Severity = "medium"    // 中等(一般bug)
    SeverityLow      Severity = "low"       // 轻微(小问题)
)
```

**四段式的核心价值**:
1. **背景(Background)**: 提供完整的场景上下文,帮助理解"为什么当时会这么做"
2. **过程(Process)**: 记录具体操作和决策依据,帮助理解"失败的路径"
3. **错误(Error)**: 详细记录问题和影响,建立"严重性认知"
4. **正解(Solution)**: 提供正确方案和预防措施,确保"知道怎么做对"

### 4. 错题本记录流程: LLM四段式引导

```
用户反馈"失败了" / "不行" / "出错了"
→ 触发错题本记录流程

LLM引导式提问(四段式):
┌────────────────────────────────────────────────┐
│  第1问: 背景信息                               │
│  "让我了解一下当时的背景:                     │
│   - 你当时想解决什么问题?                     │
│   - 在什么环境下操作的?(生产/测试/本地)      │
│   - 有哪些限制条件?(时间/资源/业务要求)"     │
│                                                │
│  用户回答后 → 构建Background结构               │
└────────────────────────────────────────────────┘
                    ↓
┌────────────────────────────────────────────────┐
│  第2问: 操作过程                               │
│  "你当时是怎么做的?                           │
│   - 具体执行了什么操作?                       │
```

不是"用户问了才召回",而是"检测到危险操作主动预警"

### 5. 进化创新: 反馈闭环 + 自动升级

不是"一次记录永久不变",而是"追踪重犯次数,自动升级规则,持续进化"

## 💡 关键讨论记录

### 讨论1: 是否需要单独实现错题本?

**问题**: 是否可以融合到反馈策略的打分机制中,而非单独存储?

**结论**: 必须单独实现ErrorBook存储

**原因**:
1. 错题本不仅是"负向评分",更是"完整的学习案例"(四段式)
2. 需要主动检测能力(trigger_patterns)
3. 需要团队共享能力(按severity筛选)
4. 需要结构化预防措施和检查清单

### 讨论2: 静态知识 vs 动态场景的矛盾

**问题**: 知识本身是静态的,如何评价其在不同场景下的相关性?

**解决方案**: 动态场景权重

```
最终得分 = 基础相关度 × 场景权重 × 类型权重

其中:
- 基础相关度: LLM评估的语义相似度(0-1)
- 场景权重: LLM动态评估与当前场景的匹配度(0-2)
  * 场景A: 0.85 × 1.8 × 2.0 = 3.06 (高分,召回)
  * 场景B: 0.85 × 0.6 × 2.0 = 1.02 (低分,不召回)
- 类型权重: 错题本×2.0, 正常记忆×1.0

→ 完美解决"静态知识 vs 动态场景"的矛盾
```

### 讨论3: Token消耗问题

**问题**: 四段式完整内容传给LLM,Token消耗会很大

**解决方案**: 两阶段传递

```
阶段1(精排序): 只传Summary (~100 tokens/条)
阶段2(最终合成): 只传TOP-N的完整内容 (~800 tokens/条)

节省效果: 51% ✅
```

### 讨论4: 四段式结构必要性

**核心观点**: 错题本不应只记录"错误",应该包含完整的背景、过程、错误、正解

**价值**:
1. Background: 提供场景上下文,帮助理解"为什么当时会这么做"
2. Process: 记录操作和决策,帮助理解"失败的路径"
3. Error: 详细问题和影响,建立"严重性认知"
4. Solution: 正确方案和预防措施,确保"知道怎么做对"
